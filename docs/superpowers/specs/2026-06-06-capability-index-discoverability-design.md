# Capability Index — Tool Discoverability Design

**Date:** 2026-06-06
**Status:** Approved (design); pending implementation plan
**Topic:** Stop agents from nearly rebuilding tools that already exist

## Problem

Agents working in this repo repeatedly get to the point of designing/building a
new tool, then discover at the last moment that an equivalent already exists.
This wastes tokens and signals a discoverability/documentation gap.

This is grounded in concrete incidents found in session history, not a
hypothetical:

- An agent spent **hours** planning an `mwcc_debug` integration (vendor fork,
  MinGW Makefile, wibo patch, wrapper scripts) before discovering mid-session
  that `RootCubed/mwcc-inspector` already existed and was strictly better.
- An agent drafted a "custom permuter scorer" design before finding
  `melee-agent debug score` already does exactly that.
- An agent planned a Ghidra cross-reference tool, then a skill-list refresh
  revealed the `/ghidra` skill was already live.
- An agent re-analyzed permuter structure-search axes that had documented
  yield-0 findings in memory.

In every costly case the capability was one `--help` or one skill refresh away.
The agents had access; they lacked the reflex to check before building.

## Root-cause analysis (ranked)

1. **Agents don't audit-first at planning time (dominant).** A behavioral gap,
   not a missing-file gap. All expensive incidents were agents in "design a new
   tool" mode that never gated on "does this exist?"
2. **Most of the surface isn't in auto-loaded context.** A fresh agent auto-sees
   CLAUDE.md (~13 of 20 skills, ~15 of 150+ CLI subcommands) and skill
   frontmatter. The entire `debug` subsystem (80+ subcommands, including the
   `debug score` that got re-proposed) is documented nowhere auto-loaded.
   ~135 of 150+ subcommands are invisible until you run `--help`.
3. **What is surfaced is stale, which is worse than missing.** CLAUDE.md's
   command list is significantly out of date, creating false confidence that
   the ~15 listed commands are the whole inventory. At least one skill doc
   references removed commands, so an agent follows it, fails, and concludes the
   capability doesn't exist → rebuilds.
4. **No single "what can I do?" entry point.** No `capabilities` command, no
   generated index. A diligent agent must union 4+ partial, disagreeing
   surfaces (CLAUDE.md, `--help`, skill files, `docs/agent-tool-manifest.md`).
5. **Tools are named by implementation, not task.** `ghidra` not `find-callers`;
   `mwcc-debug` not `debug-registers`; `debug score` not "score a candidate".
   Intent-based search doesn't resolve to the right tool.

## Decisions (from brainstorming)

- **Ambition: incremental.** Build the generated, auto-loaded, queryable index +
  an audit-first rule now. Measure. Add an *enforcing* pre-build hook only if
  near-rebuilds persist. (Evidence-first; we lack a real base-rate/cost number,
  and the index must back any future hook anyway.)
- **Coverage: CLI + skills only.** Generate from the `melee-agent` Typer command
  tree + `.claude/skills/*/SKILL.md` frontmatter. This is where every documented
  incident happened. Two clean source-of-truth inputs; highest signal-to-noise.
  Standalone `tools/*.py` scripts and superpowers skills are out of scope.
- **Auto-load: tiered.** Auto-load a compact group-level summary each session;
  full per-command detail only on demand. Directly addresses the stated
  token-cost concern.

## Grounding facts (verified, not inferred)

- **CLI framework is Typer** (`tools/melee-agent/src/cli/__init__.py:24`), which
  is built on Click → the command tree is programmatically walkable. The CLI
  half of the index can be generated by introspection and therefore *cannot go
  stale*.
- **The session hook injects context only in the remote branch.**
  `.claude/hooks/session-startup.sh:51` exits before the `additionalContext`
  block when `CLAUDE_CODE_REMOTE != "true"`. So local sessions (the common case)
  auto-load nothing today. Wiring in the tiered summary requires emitting
  `additionalContext` **unconditionally**, merged with the existing remote-env
  notice so exactly one JSON object is produced on stdout.
- **A partial solution already exists:** `docs/agent-tool-manifest.md` — but it
  is not linked from CLAUDE.md, not auto-loaded, not generated, and omits skills
  and the `debug` tree. The fix is largely "generate and wire up the thing that
  half-exists," which keeps cost low.
- **No `capabilities` command exists today** → no naming collision.

## Components

### 1. `melee-agent capabilities` command
Location: `tools/melee-agent/src/cli/capabilities.py`, registered in
`tools/melee-agent/src/cli/__init__.py` as `capabilities_app`.

- `capabilities search <task>` — **live** introspection of the root Typer app
  (walk sub-apps → commands, pulling each command's `short_help`/docstring) +
  parse `.claude/skills/*/SKILL.md` frontmatter (`name`, `description`). Rank
  results by keyword match against name + help text, boosted by a curated
  **task-alias map** (see below). Output: `name — one-liner — how to invoke`.
  Always fresh (computed from source).
- `capabilities show [group]` — live full detail for a group (or all groups +
  all skills).
- `capabilities generate` — write the two committed artifacts (below).

**Task-alias map:** a small hand-maintained table (in-repo, in the module or an
adjacent data file), ~10–15 entries to start, mapping task intent → capability
id, to bridge the implementation-vs-task naming gap. Seed entries:
- `find callers` / `cross reference` → `ghidra` skill, `commit check-callers`
- `debug registers` / `register allocation` → `mwcc-debug`, `mwcc-inspect`
- `score candidate` / `scorer` → `debug score`
- `per-file progress` / `per-file stats` → `extract files`
- `diff two report.json` / `compare matches` → `compare_branch`, `diff_changes`
- `find similar functions` → `opseq` skill, `patterns similar`, `debug search`

**No-match behavior:** when search finds nothing, return
"no existing capability found for `<task>` via search — it may be safe to build;
double-check `--help` of the nearest group first." Never assert a capability
"doesn't exist" (mirrors the project's never-claim-unmatchable discipline —
report "not found via search," not "absent").

### 2. Two generated artifacts (committed)
- `.claude/capabilities-brief.md` — the **compact tier**: one line per CLI group
  listing its subcommand verbs (e.g. `debug: dump, inspect, target, suggest,
  mutate, intervene, permute, util, search, score`), plus one line per skill.
  This is the only artifact auto-loaded at session start. Kept small.
- `docs/CAPABILITIES.md` — full per-command/per-skill detail; the `show` target
  and human-browsable inventory. **Absorbs `docs/agent-tool-manifest.md`**; the
  old file is reduced to a one-line pointer to `docs/CAPABILITIES.md` so any
  existing references keep resolving.

### 3. Session-hook change
`.claude/hooks/session-startup.sh`: emit `additionalContext` **unconditionally**
(before the remote-only `exit 0`), injecting the contents of
`.claude/capabilities-brief.md` plus a one-line nudge:
"Before building any tool/script/command, run `melee-agent capabilities search
<task>` — the inventory is large and your need may already exist."
Merge this with the existing remote-environment notice so the hook still emits a
single JSON object. The hook only `cat`s the static brief (no Python at startup
→ fast); if the brief file is missing, degrade gracefully (skip the capabilities
block, still emit the remote notice).

### 4. Audit-first rule (incremental lever for cause #1)
Add a short, prominent block to:
- the top of `CLAUDE.md` (auto-loaded), and
- the `decomp` and `workflow` skills.

Content: "Before building a new tool/script/command, `capabilities search` first.
There are 150+ CLI subcommands and 20 skills; assume your need may already
exist." Include the real cautionary examples (mwcc-inspector, `debug score`).

### 5. Freshness guard (kills cause #3)
A pre-commit hook + CI step runs `capabilities generate` and **fails on drift**
between the committed artifacts and what regeneration produces. Ensure both
generated artifacts and the hook edit are part of the fork-tooling overlay that
`sync-upstream` preserves (per the known `configure.py`-clobber issue) so they
are not silently reverted on upstream sync.

### 6. Measurement (decision criterion for phase 2)
Lightweight, since auto-instrumentation is hard:
- A usage counter for `capabilities search` in the agent state DB
  (`~/.config/decomp-me/agent_state.db`).
- A periodic grep of recent session transcripts for the rebuild pattern (the
  phrases used by the audit: "already exists", "turns out there", "already
  built", "instead of building", etc.).

If near-rebuilds persist despite the index and the rule, that is the trigger to
build the enforcing pre-build hook (phase 2). Until then, the hook is not built.

## Data flow

```
session start ─► hook cats capabilities-brief.md ─► injected as additionalContext (every session)
agent considers building X ─► audit-first rule nudges ─► `capabilities search X`
        ─► live result from Typer tree + SKILL.md ─► uses existing tool (no rebuild)
commit ─► pre-commit/CI runs `capabilities generate` ─► fails on drift ─► artifacts stay fresh
```

## Error handling

- **Session start must never break.** The hook only reads a static file; missing
  file → skip the capabilities block, still emit the remote notice; never exit
  non-zero in a way that blocks the session.
- **Search no-match** → "not found via search" guidance, never "doesn't exist."
- **Typer introspection failure** in `capabilities` → command errors through the
  existing `ReportingTyper` path (prints a copyable `issue report` command).

## Testing

- **Generator unit tests:** walking a known Typer app fixture yields the
  expected command entries; `SKILL.md` frontmatter parsing; task-alias
  resolution.
- **Snapshot/drift test:** `capabilities generate` output is stable; the
  drift-check correctly fails when source changes without regeneration.
- **Search-relevance regression (tests the actual failures):** each documented
  incident query — "scorer", "find callers", "register allocation", "per-file
  progress", "diff report.json" — returns the correct tool in its top results.

## Out of scope (YAGNI)

- The enforcing pre-build hook (deferred to a measured phase 2).
- Indexing standalone `tools/*.py` scripts.
- Indexing/curating the 70+ superpowers/global skills.
- Embeddings or fuzzy ranking beyond keyword + curated aliases.
- Consolidating/deprecating the genuinely overlapping commands the audit found
  (`scratch compile` vs `scratch update`; `extract get --create-scratch` vs
  `scratch create`; `scratch update-context` vs `scratch update`). Noted as a
  follow-up; the index will surface them rather than resolve them.

## Open risks

- **Compact-tier token budget.** The brief must stay small enough that
  auto-loading it every session is clearly cheaper than the rebuilds it
  prevents. Group-level granularity is chosen for this reason; revisit if the
  brief grows.
- **Compliance is assumed.** The incremental plan relies on agents actually
  running `capabilities search`. Measurement (#6) exists precisely to detect if
  this assumption fails and phase 2 is needed.
- **Surfacing redundancy.** Faithfully listing overlapping commands may add
  confusion; mitigated for now by the curated alias map pointing intent at the
  preferred tool.
