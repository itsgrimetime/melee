# Remote permuter runner design

## Goal

Add tooling that makes it easy to run decomp-permuter jobs on existing
Coder/cloud instances with many CPU cores, then sync the results back to the
local Melee worktree for the existing triage workflow.

The instances are already provisioned. They may be idle, but SSH connection is
the wake mechanism, so this feature must not depend on Coder APIs.

## Non-goals

- Do not create, resize, or destroy Coder instances.
- Do not fork or patch decomp-permuter as part of this feature.
- Do not require copying the full Melee repository or build output on every
  run.
- Do not make upstream PR descriptions or source changes mention local remote
  runner details.

## Command shape

Add a subgroup under the existing permuter CLI:

```bash
melee-agent debug permute remote targets
melee-agent debug permute remote submit -f FN --target coder64 [--threads N] [--mode stock|mwcc]
melee-agent debug permute remote list
melee-agent debug permute remote status JOB
melee-agent debug permute remote tail JOB
melee-agent debug permute remote fetch JOB [--triage]
melee-agent debug permute remote stop JOB
```

`submit` is detached by default. It starts the remote job and returns a job id
plus the important local and remote paths. Follow-up commands operate on that
job id.

## Configuration

Store target configuration in the local decomp-me config directory:

```toml
[target.coder64]
ssh = "coder.coder64"
remote_melee_root = "/home/coder/melee"
remote_perm_root = "/home/coder/decomp-permuter"
threads = 64
session_prefix = "melee-perm"
```

This file is local-machine state, not repo state. It can include several
targets. The CLI should print a clear message if no config exists, including
the expected path and a minimal example.

## Submit Flow

1. Resolve the local permuter function dir using the same search behavior as
   existing `debug permute` commands.
2. Ensure the local permuter dir is ready enough to run remotely:
   - `settings.toml` exists or can be generated by existing config helpers.
   - `compile.sh` has been fixed by existing mac-compatible fix helpers where
     applicable, while preserving portability to Ubuntu.
3. Sync the local function dir to a job-specific remote working copy:
   `<remote_perm_root>/remote-runs/<JOB>/nonmatchings/<FN>/`.
4. Sync the minimal Melee source/config state needed by the remote runner:
   - The first version should prefer explicit `rsync` of dirty source/config
     files over copying build output.
   - If the remote repo is missing or not on the requested revision, fail with
     a setup checklist instead of trying to provision everything.
5. Start a detached remote job in `tmux`:
   - Stock mode: `./permuter.py remote-runs/<JOB>/nonmatchings/<FN> --threads <N>`.
   - MWCC mode can use the existing `permute_with_mwcc.py` wrapper only when
     the remote host has the required setup.
6. Write metadata locally under:
   `~/.config/decomp-me/permuter-jobs/<job-id>.json`.
7. Write metadata remotely in the run directory so jobs can be inspected even
   if local metadata is stale or absent.

Job ids should include function, target, and timestamp, for example:

```text
mnFoo_80212345-coder64-20260525-143012
```

## Fetch Flow

`fetch JOB` copies remote results back without stopping the job:

- `output-*` candidate directories.
- `best.c` or equivalent winner files if present.
- stdout/stderr logs.
- remote metadata.

By default, fetched results should land in a local run-specific directory under
the function's permuter dir, avoiding accidental overwrite of local sessions.
`--triage` prints the local `melee-agent debug permute triage` command to run
against the fetched candidate directory.

Fetch must be idempotent so it is safe to run repeatedly while the remote job is
still active.

## Status, Tail, Stop, and List

- `targets`: print configured targets and basic reachability if requested.
- `list`: show local job metadata with job id, function, target, state, and
  last known remote path.
- `status JOB`: SSH to the remote, check the tmux session and run metadata, and
  print whether the job appears active.
- `tail JOB`: stream the remote log. It should work even after the tmux session
  exits if the log file remains.
- `stop JOB`: stop the remote tmux session for that job without deleting
  outputs.

## Portability Requirements

Local development runs on an arm-based macOS machine. Remote jobs run on Ubuntu.
The implementation must avoid assuming that the same shell paths, binary names,
or platform-specific wrappers exist on both sides.

Remote command construction should use conservative POSIX shell syntax and
quote paths. Local tests should run on macOS without a live SSH target by
injecting fake command runners for `ssh` and `rsync`.

The remote runner should not assume macOS-only helpers such as wibo paths unless
the user explicitly selects a mode that requires them and the preflight confirms
they exist remotely.

## Error Handling

- SSH failure: identify the target and operation, and show the failing command
  only when useful.
- Missing remote repo or permuter root: print a setup checklist with the
  expected paths from config.
- Missing local permuter dir: reuse the existing import hint from `debug
  permute`.
- Missing `tmux`: fail clearly. A `nohup` fallback can be added later, but the
  first version should keep process management predictable.
- Remote job already exists for the same id: refuse to overwrite metadata unless
  a future `--force` option is added.

## Implementation Boundaries

Create a small remote-runner module rather than putting all orchestration in
`cli/debug.py`. The CLI layer should parse options and print results; the module
should own config loading, job metadata, command construction, and command
runner injection.

Suggested modules:

- `src/mwcc_debug/permuter_remote.py`
- Tests in `tools/melee-agent/tests/test_mwcc_debug_permuter_remote.py`

The implementation should reuse existing permuter path resolution and import
hints from `cli/debug.py` where practical, but avoid making those helpers more
global than necessary unless tests justify it.

## Testing

- Config parser accepts valid target files and reports missing or malformed
  files.
- Submit builds the expected `rsync`, `ssh`, and `tmux` commands with paths
  safely quoted.
- Submit records local metadata.
- Status handles active, missing, and completed tmux sessions.
- Fetch builds the expected `rsync` command and can optionally request local
  triage.
- CLI smoke tests cover help text and the common missing-config/missing-target
  errors.

No tests should require a real Coder instance, SSH connection, or remote
decomp-permuter checkout.
