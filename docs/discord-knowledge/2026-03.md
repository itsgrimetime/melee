## Compiler Patterns

### MWCC Debug Listing / Compiler Internals Discovery
A major discovery was made: MWCC has built-in debugging functionality (`#pragma debuglisting on`) that can be activated via a patched DLL to dump extensive information about every optimization decision the compiler makes:
- Dumps IR and assembly at every pass (80+ passes typically)
- Shows interference graphs and how virtual registers map to physical registers
- Reveals register importance/weighting that determines allocation priority
- Shows propagatable expression assignments and dead code elimination decisions
- Loop unrolling decisions are logged extensively
- Can dump each function's information and snapshots at each step of the pipeline

Key caveats:
- Cannot run with full ninja builds -- spawning many instances causes file handle conflicts via the sjis loader
- Must swap in the patched DLL for single-TU compilation and swap back out
- Output can be 100MB+ of text for a single TU
- The 7.0 compiler internals are closer to 1.2.4n than expected

Tool: https://github.com/Savestate2A03/mwcc_debug
- Source: savestate (2026-03-13 through 2026-03-14)

### Register Importance and Allocation
From the debug listing output, MWCC assigns an "importance" score to registers. A register that is "too important" to move down in the allocation order cannot be swapped by simple declaration reordering. You need to distribute the importance by restructuring variable usage so the compiler allows the swap.
- Source: savestate (2026-03-14)

### ABS Macro Triple-Expansion Bug
When using `ABS()` with an assignment expression inside, the macro expands the argument three times:
```c
// This:
fp->mv.ys.specials.x1C = ABS(fp->gr_vel = fp->self_vel.x);
// Expands to:
fp->mv.ys.specials.x1C = ((fp->gr_vel = fp->self_vel.x) < 0
    ? -(fp->gr_vel = fp->self_vel.x)
    : (fp->gr_vel = fp->self_vel.x));
```
The assignment executes up to 3 times in the expanded form. This was likely intentional by the original developer who didn't realize the macro's triple expansion. The decompiler (m2c/decomp.py) sees multiple stores to `gr_vel` and loads from `self_vel.x`.
- Source: lukechampine (2026-03-19)

### Preventing Auto-Inlining via Wrapper Functions
When MWCC auto-inlines a function you don't want inlined, one technique is to increase the call depth by wrapping it in more functions, or make the inlined function bigger so it can't be efficiently inlined. Another approach is moving the function body into a separate inline helper.
- Source: ribbanya (2026-03-02)

## Matching Techniques

### HAL Float Temp Pattern (f0, f1)
HAL developers had a tendency to use `f0, f1` float temporaries. When you see floats being reused in multiple places, using the same float variables for multiple operations can eliminate extra stack variables:
```c
// Reusing the same float values across operations can reduce stack usage
// instead of declaring separate float vars
```
This pattern shows up frequently in sysdolphin code.
- Source: werewolf.zip (2026-03-10)

### Inline Scope for Float and Stack Ordering
When float ordering of `slope` and `intercept` affects stack usage, creating a new scope (braces block) in the middle of a function allows declaring variables mid-function, which can fix stack/float ordering issues:
```c
void func(...) {
    // ... earlier code ...
    { // new scope
        f32 slope = ...;
        f32 intercept = ...;
        // use them here
    }
    // ... later code ...
}
```
- Source: werewolf.zip (2026-03-10)

### Permuter for Register Swaps
The decomp-permuter tool is highly effective for solving register swap mismatches:
- Run for 1-10 minutes per function
- Iteratively running (applying best result, then re-running) often yields better results than a single long run
- Common permuter fixes: reordering variable declarations, adding temporary variables, chained assignment patterns
- Typical permuter output creates `new_var` temporaries that move arguments through an extra assignment
- 40-50 small functions can be matched in an hour of permuter runs
- Only accept 0-score (100% match) results from mass-permuting; partial improvements can introduce regressions

Setup: https://github.com/jellejurre/decomp-permuter

Caveats:
- Permuter diffs are for the preprocessed "big file" and can't be directly applied to source
- Results can create "fake" improvements where score drops but code is semantically wrong
- Some register swaps are not permuter-fixable (require understanding compiler register weighting)
- Source: jellejurre, jurrejelle (2026-03-02 through 2026-03-19)

### Permuter Discord Bot
A Discord bot (`JellyBean`) was created for running the permuter remotely:
```
!permute [fork_name] [branch_name] [function_name] [timeout_in_minutes]
```
Example: `!permute doldecomp master ftKb_SpecialNFx_800FE240 1`
- Maximum timeout: 10 minutes
- One run at a time (queued)
- Bot must share a server with the user to DM results
- Source: jellejurre (2026-03-05 through 2026-03-06)

### Stack Padding: Initialized vs Uninitialized Arrays
When linking, be careful about initialized vs uninitialized stack arrays:
- `u8 _2[0x10] = { 0 };` generates data in `.rodata` section (bad for linking)
- `u8 _2[0x10];` does not generate `.rodata` data
- If you need the initialization for stack offset generation, use individual `u32 pad1 = 0; u32 pad2 = 0;` variables instead
- Source: jellejurre (2026-03-19)

### Data Matching Guidance for Incomplete Files
When a file is not yet fully matched, don't try to match data ordering:
- Don't declare static strings at the top of the file
- Use `HSD_ASSERT` directly rather than declaring the assert strings separately
- Use `__FILE__` for filename strings instead of declaring them manually
- Use inline string literals: `OSPanic(__FILE__, 0x459, "thing")` instead of referencing static string variables
- Data ordering and const float reordering via inlines only matters when actually linking the file
- Source: werewolf.zip (2026-03-18)

## Type Information

### Stage Internal IDs Correction
Several internal stage IDs were found to be wrong in both the decomp and community tools (MEx):
- There is no `GRKIND_ADVKRAID` -- it's `GRKIND_KINOKOROUTE` (Mushroom Kingdom)
- `GRKIND_ADVTE` is an undocumented stage ID (purpose unknown)
- Assuming 0=DUMMY, 23=ICETOP, 26=AKANEIA
- MEx's stage IDs were reported as incorrect on their issue tracker

## Tool Tips

### Build Validation Commands
To catch CI failures locally before pushing:

```bash
# Build with prototype requirements (matches CI behavior)
python configure.py --require-protos
ninja

# Check for context build failures
tools/m2ctx/m2ctx.py -pqx && melee-mwcc -nofail build/ctx.c -v -o build/ctx.c.o

# Create symlink for compiler if missing
ln -s build/compilers tools/mwcc_compiler

# Check for regressions
ninja diff

# Clean build
ninja -t clean && ninja

# Check issues (requires clang dev packages)
cargo run -rpq melee-issues
```

Note: `--require-protos` ensures functions have prototypes before calls, matching CI behavior. Without it, builds may succeed locally but fail in CI.
- Source: ribbanya, jellejurre (2026-03-01, 2026-03-21)

### ninja apply vs dtk dol apply
`ninja apply` and `dtk dol apply` are different commands that can conflict. Running `dtk dol apply` can undo changes made by `ninja apply`. Use `ninja apply` consistently.
- Source: werewolf.zip, ribbanya (2026-03-04)

### Linking Workflow Steps
When all functions in a file are 100% matched and you want to link:
1. Remove unused imports (compile to verify each removal)
2. Adjust offsets, alignment, and types of symbols in `symbols.txt`
3. Create data structures to match objdump from objdiff
4. Adjust stack pads, fill up data needed as padding
5. Match labels in symbols file with actual usage
6. Adjust whether data is const or not based on read-only status
7. Run `ninja apply` to rename auto-generated symbols (e.g., `@234` style)
- Source: jellejurre, werewolf.zip (2026-03-19 through 2026-03-20)

### Linking: String Padding Between Data Symbols
When linking, if you see padding issues between string data symbols:
- The linker auto-pads data sections to 8 bytes
- Don't manually pad strings with `\0\0\0\0` in the declaration
- Instead, just use the string inline and let the compiler/linker handle padding
- For end-of-section padding in `splits.txt`, add `-N` (e.g., `-3` for 3 bytes)
- Source: werewolf.zip, jellejurre (2026-03-20)

### decomp.me Function Name Mismatch
When using decomp.me, verify that function call labels match the actual addresses. A function called `it_8026BD84` is different from `it_802efd84` -- getting the address wrong in a prototype can cause unexpected inlining behavior.
- Source: mariolover64, jurrejelle (2026-03-02)

### decomp.dev API for Progress Visualization
decomp.dev exposes an API for generating treemap images at any commit:
```
https://decomp.dev/doldecomp/melee/GALE01/<commit_hash>.png
https://decomp.dev/doldecomp/melee/GALE01/<commit_hash>.svg
```
Supports size parameters: `?w=1900&h=950`
- Source: encounter (2026-03-17)

## Pitfalls and Gotchas

### Mass Permuter Risks
When mass-permuting functions:
- Improved match score does not mean correct code -- a function can go from 98% to 99% while introducing semantically wrong code (e.g., using the wrong variable in one spot)
- Only accept 100% matches from automated runs
- Permuter can find local minima that make future manual matching harder
- Some permuted code is "atrocious" and harder to clean up than starting from m2c output
- Always review permuter output; `new_var` assignments and dead code like `if(a->x4 && !a->x4) {}` or `!new_var;` are common artifacts
- Source: ribbanya, jurrejelle (2026-03-02 through 2026-03-15)

### Pointer Math in PRs
Raw pointer arithmetic and casts into structs should be avoided in PRs when possible:
- Better approach: modify structs/unions to add proper fields
- If the struct context isn't available, pointer math may be temporarily acceptable but should be flagged for cleanup
- AI agents are particularly prone to introducing pointer hacks when they can't find proper struct definitions
- Source: jurrejelle, ribbanya, werewolf.zip (2026-03-12)

### AI Agent Collision on Easy Functions
Multiple AI agents working on "easy_funcs" across the codebase leads to duplicated work:
- Working on individual files (TUs) and marking them via draft PRs helps avoid collision
- Checking open PRs via regex is brittle
- The TU wiki table can help but doesn't cover cross-TU easy function sweeps
- Some contributors are working on server-based function assignment to solve this
- Source: winstax, jurrejelle, lukechampine (2026-03-17 through 2026-03-19)

### Build Failures from Cyclic Redundancy Checks
If you see errors like:
```
sjiswrap: Failed to read file D:\melee\src\melee\if\ifmagnify.h
Data error (cyclic redundancy check). (OS error 23)
```
This is likely a corrupted local build state. Fix: fresh clone. `ninja -t clean` and deleting build directories may not be sufficient.
- Source: werewolf.zip (2026-03-04)

### Don't Trust Community Documentation Blindly
MEx and other community tools can have incorrect data (e.g., wrong stage IDs). Always verify against the actual game binary.
- Source: werewolf.zip (2026-03-11)

## AI/LLM Usage Notes

### Opus 4.6 for Decompilation
Multiple contributors report Opus 4.6 as the strongest model for decompilation work:
- Very token-hungry but produces better results than alternatives
- One contributor ran it for 36+ hours straight before being rate-limited
- Can "one-shot" an 8KB file in a few hours with iteration
- Cost can be reduced significantly using GitHub Copilot Pro ($10/month for ~100 Opus requests)
- Guide: https://github.com/jurrejelle/ai-melee-decomp
- Source: jurrejelle, stephenjayakar (2026-03-07 through 2026-03-13)

### AI Limitations
- AI struggles more as easy functions are consumed; larger/harder functions remain
- AI agents frequently introduce pointer math hacks instead of proper struct modifications
- AI-generated names and comments can be hallucinated -- always verify
- For linking, AI is currently above its capability level (requires too much iterative context)
- The mwcc debug output is "really hard for [agents] to interpret into anything useful" without additional tooling on top
- Source: encounter, jurrejelle, aizk (2026-03-14 through 2026-03-20)
