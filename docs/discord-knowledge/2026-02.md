# Decompilation Knowledge: 2026-02

This document distills useful decompilation knowledge from Discord conversations in the #smash-bros-melee channel during February 2026.

---

## Compiler Patterns

### Sysdolphin vs Melee Bitfield Usage
Sysdolphin structs do not use bitfields; they use raw bit operations on ints. Melee structs do use bitfields. Using bitfields in sysdolphin code will produce different codegen and fail to match.
- Both were developed by HAL, but different programmers had different habits
- Example: `jobj->flags` does not use a union bitfield
- Source: rtburns (2026-02-26)

### Compiler Array Access Optimization
The compiler can optimize "access array in loop by index" into "increment the pointer each loop." When you see pointer arithmetic in m2c output that looks unnatural, try undoing the optimization and writing a simple indexed loop instead.
```c
// Instead of pointer arithmetic from m2c:
// *ptr++ = value;

// Write natural indexed access:
array[i] = value;
```
This often instantly resolves mismatches.
- Source: altafen (2026-02-07)

### Loop Unrolling by MWCC
When a loop body is small enough, the compiler will unroll it. If you see repeated identical operations in the asm with slightly different offsets (e.g., accessing struct fields at 0x100, 0x1A8, 0x250), it is likely a loop over an array of structs that the compiler unrolled.
- Write the code as a simple loop; the compiler handles unrolling
- Source: kellz (2026-02-22)

### crclr cr1eq Instruction
The `crclr cr1eq` instruction is generated by the compiler when calling a variadic function (e.g., `printf`, `efAsync_Spawn`). The cr1eq bit denotes whether there are any float arguments. Either pass a float on the argument list or don't, and the compiler will generate the appropriate instruction. Make sure the function prototype is visible.
- Source: ieee802dot11ac, altafen, gamemasterplc (2026-02-22)

### int vs u8 in Function Parameters
The compiler reduces `int` down to `u8` where convenient. `s32` is typedef'd to `long` on the GC architecture and is NOT subject to that reduction. When in doubt, use `int` as the default parameter type and see if it matches. The decompiler (m2c) outputs `s32` by default, which often needs to be manually replaced.
- `bool` is a typedef of `int`, and you can use `true`/`false`
- Source: ribbanya (2026-02-21)

### Stack Size Changes from Unused Parameters
Adding an unused parameter to a function can increase the stack size by 8 bytes, causing a mismatch. If a parameter appears unused, verify whether it should actually exist by checking call sites.
- Source: jellejurre (2026-02-21)

## Matching Techniques

### M2C Output as Starting Point
M2C (the "Decompilation" button on decomp.me) gets very close to matching code. Errors in its output are usually type mismatches or incorrect struct usage, which are much easier to fix than writing from scratch. Always start from m2c output rather than manually translating assembly.
- Source: kellz, ribbanya (2026-02-07)

### Switch Statement Detection
When m2c or manual decompilation produces a chain of `if/else if` comparisons like:
```c
if (val == 2) {
    r6->xC = r4;
} else if (val >= 2) {
    if (val < 4) {
        r6->xC = r3;
    }
} else if (val >= 1) {
    r6->xC = r5;
}
```
This is usually a switch statement. The key indicator is "sandwiched branches" in the asm.
- Source: kellz (2026-02-07)

### Scope Blocks for Register Allocation
MWCC only allows variable declarations at the start of a block. To move declarations lower in a function (which can help sort out register allocation), use scope blocks:
```c
void func(void) {
    // early code...

    {
        Type* var = something;
        // code using var
    }

    // later code...
}
```
This is an intuitive process: if a variable is only used in one part of a function, moving its declaration closer to its usage may fix register mismatches.
- Source: ribbanya (2026-02-22)

### Bitfield Type Sign for extsb
The `extsb` (extend sign bit) instruction indicates a signed type. If you see `rlwinm` and `srawi` (bit shift operations), the code uses bitfields. Changing a bitfield from `u8` to `s8` can fix a mismatch when `extsb` is present.
- Example: changing `b0` field from `u8` to `s8` achieved 100% match
- Source: ribbanya (2026-02-21)

### Fixing M2C_UNK Declarations
`M2C_UNK` in function signatures means the return type and arguments are unknown placeholders. Changing these to correct types is expected and necessary. Check call sites and struct definitions to determine correct types.
- Source: werewolf.zip (2026-02-17)

### Passing Struct Pointers: Address-of Fields
When a function takes a `u8*` parameter but the call site passes `&struct->field` where field is a `u8` (not a pointer), you may need to cast the field's address to the appropriate struct pointer type.
- Source: elcompadiego, altafen (2026-02-06, 2026-02-07)

### Stack Padding from Missing Inlines
When the compiled function has extra stack space compared to the target, it usually means a missing inlined function or variable, not that you need explicit padding. Look for inline candidates before adding `PAD_STACK`.
- Source: ribbanya (2026-02-08)

### Cursed Ternary for Loop Unrolling Match
Sometimes a ternary expression, even if it looks unusual, is the only way to match unrolled loop code:
```c
// "sort of cursed ternary but its 100%"
```
The compiler may produce code from ternaries that would be impossible to replicate with if/else.
- Source: kellz (2026-02-22)

## Type Information

### Classic Mode Splash Screen Struct
```c
static struct {
    int x0;
    HSD_JObj* x4[0xF];
} lbl_804735A8;

static struct {
    HSD_ImageDesc x40[3];
    HSD_ImageDesc x88[3];
    u8 pad[0xE0 - 0x88 - 3 * 0x18];
    u8 xE0;
    u8 xE1;
    u8 xE2;
    u8 xE3;
    int xE4; ///< related to model scale
    int xE8; ///< game type
    u8 xEC;
    u8 xED;
    u8 xEE; ///< roadmap (IrRdMap) progress
    u8 xEF; ///< left char amt
    u8 xF0; ///< right char amt
    u8 xF1[3]; ///< left char ids
    u8 xF4[3]; ///< right char ids
} lbl_804735E8;
```
- Source: kellz (2026-02-02)

### Name Entry Slots
- Maximum of 120 name entry slots (indices 0x00 to 0x77)
- Index 0x78 (120) is hardcoded as an invalid/null nametag slot
- The `x4` field in the classic mode init struct is `max_slots` or current name entry slot
- Source: kellz, altafen, elcompadiego (2026-02-02)

### Struct Alignment and Padding
Structs are aligned by the compiler unless explicitly packed. If a struct array ends at an odd offset (e.g., 0x446), the next field will start at the next aligned address (e.g., 0x448). You do not need to add manual padding between struct members for alignment.
- Source: kellz, jurrejelle (2026-02-22)

### lbArqGlobal Struct Layout
`lbArqGlobal` at `0x804316C0` is a 0x1F0-byte BSS struct. The `lbArqNode* list[3]` at offset 0x1E0 contains pointers, while the pad region contains actual `lbArqNode` structs (not pointers). The distinction between embedding a struct vs storing a pointer to it matters for memory layout.
- Source: elcompadiego, amber_0714 (2026-02-18)

### DolphinSDK Types Are Fixed
The DolphinSDK function signatures and types should never be changed, as they were matched using actual debug symbols. When SDK functions take `u32` for pointer arguments, you must cast your pointers to `u32` rather than modifying the SDK headers.
- Example: `ARQPostRequest` parameters should not be changed
- Source: werewolf.zip (2026-02-20)

## Function-Specific Insights

### Character Unlock System
- `gmMainLib_8015ED8C` retrieves the unlocked character bitmask
- `gm_80164840` checks if a specific character is unlocked
- `gm_80164ABC` checks if all characters are unlocked
- `gm_80164600` checks if all stages are unlocked
- Unlock functions are concentrated in `gm_1601`
- Memory card data starts at `0x8045BF28`, ends at `0x8046B0EC`
- Source: ribbanya, werewolf.zip (2026-02-02, 2026-02-17)

### HomeBros Exploit
The name entry read function does not validate input length. Writing 0xD4 bytes of garbage overwrites the stack, followed by the original return address (804EE8F8) and a jump target. This gives full control via a pre-setup memory card.
- Source: werewolf.zip (2026-02-02)

### Classic Mode Configuration
- `gm_801B34B8_OnInit` sets default classic mode configuration (stocks, name entry, color, difficulty)
- The game has 11 classic mode phases (including 3 bonus phases at positions 3, 6, and 9)
- Bonus stages set stocks to 1; if you "fail," you respawn once, then the second death ends the stage
- Source: elcompadiego (2026-02-02, 2026-02-07)

### Pokemon Event Match Field
- `gm_8016AEB8` returns a field from `lbl_8046B6A0.x24C8.x28` (memory address `0x8046DB90`)
- Returns 1 for Event Match #37 "Legendary Pokemon," 0 for normal matches
- Manually setting this to 1 in non-event matches causes a game freeze
- Source: elcompadiego (2026-02-23)

### Menu System Architecture
Menus track the currently highlighted entry as an integer. The menu's think function checks which entry is selected when A is pressed, then updates the Scene Major/Minor accordingly.
- Source: werewolf.zip (2026-02-02)

### GX FIFO Writes in Particle System
Manual `GXWGFifo` writes should be replaced with actual function calls from the Dolphin SDK:
- `GXPosition3f32` for 3 float writes to the FIFO
- The ordering of writes depends on the active vertex format
- Header: `extern/dolphin/include/dolphin/gx/GXVert.h`
- Source: encounter (2026-02-12)

### Item Vars Across Items
Items may share union fields (like `foxblaster.xDFC`). If an item doesn't have a field at a given offset, you should add it to that item's ItemVars struct. Cross-item typing assumptions can be incorrect when working without full file context.
- Stage GroundVars can have 2+ variants (e.g., Corneria has 3: stage, arwing, smash taunt)
- Source: werewolf.zip (2026-02-16)

### FighterOverlay May Be Fake
`FighterOverlay` in ftCommon may be fabricated. The data should instead be accessed through `fp->mv.co`. LLMs tend to prefer FighterOverlay because it is in the same file and does not require modifying structs.
- Source: altafen (2026-02-08)

## Tool Tips

### decomp.me M2C Integration
The m2c decompiler is integrated directly into decomp.me as the "Decompilation" button. Its output is a much better starting point than manually translating assembly. Fix type errors and struct issues in the m2c output rather than rewriting from scratch.
- Source: ribbanya, kellz (2026-02-07)

### Finding Cross-References with ripgrep
After building with `ninja`, disassembled asm appears in `build/GALE01/asm/`. Use ripgrep to find all call sites:
```bash
rg --ignore-vcs --vimgrep -a -w "function_name" build/GALE01/asm
```
Run from the repository root. In vim, use `gF` on output lines to jump to the file.
- Source: ribbanya (2026-02-17)

### ninja baseline/changes Workflow
```bash
ninja baseline     # Run first on clean master
ninja changes      # Shows regressions (what broke)
ninja changes_all  # Also shows improvements
```
Note: `ninja` alone only fails if a linked file breaks. Use `baseline`/`changes` for unlinked file regression detection.
- Source: altafen (2026-02-07)

### Context Generation Site
The context generation site is pinned in the channel. Use it to generate proper context for decomp.me scratches:
- https://doldecomp.github.io/melee/ctx.html
- Copy from there and paste into the decomp.me context tab
- Source: altafen (2026-02-19)

### tools/scaffold.py for Function Comments
The function comments listing addresses and sizes in source files are generated by:
```bash
python tools/scaffold.py
```
These are not written manually.
- Source: ribbanya (2026-02-20)

### easy_funcs.py Arguments
```bash
tools/easy_funcs.py -s 64 -S 512 -M 0 melee/lb  # Find unmatched functions 64-512 bytes in lb
tools/easy_funcs.py -m 80 -M 99                    # Find partially-matched functions
tools/easy_funcs.py -o                              # Names only output
tools/easy_funcs.py -a                              # Sort by address
```
- Source: ribbanya (2026-02-21)

### Swapping DOL in Dolphin
Every `ninja` run builds `build/GALE01/main.dol`. You can swap this into Dolphin directly, or set up Dolphin to boot from a directory and symlink the built DOL.
- Source: ribbanya (2026-02-22)

### Permuter Setup for Melee
For projects using Ninja instead of Make, add `permuter_settings.toml` in the project root:
```toml
build_system = "ninja"
```
A Melee-specific permuter fork with setup guide: https://github.com/jellejurre/decomp-permuter
- Source: jellejurre, jurrejelle (2026-02-26)

### objdiff on Windows
If objdiff shows no visible text on Windows, delete `%APPDATA%\objdiff` to reset configuration.
- Source: encounter (2026-02-13)

### clangd for VSCode
Install the clangd extension for VSCode instead of using the built-in MS language server. It provides much better code intelligence for the project's build configuration.
- Source: ribbanya (2026-02-22)

### configure.py --require-protos
Use `python configure.py --require-protos` to catch missing function prototypes. The GitHub CI builds use this flag.
- Source: ribbanya, altafen (2026-02-07, 2026-02-21)

## Pitfalls and Gotchas

### Matching vs Linking vs File Matching
- `ninja` only fails if a **linked** file breaks
- A 100% function match on decomp.me does not guarantee the data also matches
- "Matched file" = every function is 100% AND data matches AND it links AND passes checksum
- "Equivalent" = data matches and should link, but has a minor regswap/scheduling issue that doesn't affect behavior
- You cannot mark a TU as matching if data sections are not 100%, as the build will link in the non-matching object
- Exception: data size padding differences (pad to 8 bytes by linker) are OK
- Source: rtburns, blu9849 (2026-02-08)

### decomp.me Permuter vs Other Tools
The decomp.me permuter may report 100% when other tools (objdiff, the PR bot) show a different score. This can happen when data doesn't match, data is in the wrong section, or the data section is incomplete.
- Source: tsanummy, blu9849 (2026-02-01)

### Think Like a Developer, Not the ASM
When m2c produces a weird do-while loop with pointer arithmetic, ask "would a human write that?" HAL developers wrote natural code with for loops and simple array indexing. Pointer-increment loops in the decompiler output are usually compiler optimizations.
- HAL is not against pointer arithmetic for linked lists (`for(; thing != NULL; ++thing)`)
- But complex pointer arithmetic on arrays is a compiler optimization
- Source: revosucks, cortex420, werewolf.zip (2026-02-08)

### LLM Agent Pitfalls
- Agents uploading scratches to decomp.me without claiming them (anonymous junk)
- Agents generating matches without semantic context (raw pointer offsets instead of struct fields)
- Agents preferring incorrect but easier-to-use types (FighterOverlay vs mv.co)
- LLM-generated function names and comments are unverifiable; only match % is verifiable
- Large functions may not fit in LLM context windows; instruct agents to work on sections
- Compaction (context window summarization) loses initial context after 3-4 rounds; use file persistence for plans and directives
- Source: various (2026-02-07 through 2026-02-10)

### BSS Data Interpretation
`.bss` sections with `.skip 0xNNN` are uninitialized data filled at runtime. You must follow the initializer logic to determine the actual type. Usually it is a struct containing mutable state.
- Source: ribbanya (2026-02-02)

### Localization File Formats
- `.usd` files: English/US localization (e.g., `SdPrize.usd` contains unlock/prize messages)
- `.dat` files: Japanese localization (and sometimes contain English too)
- Special message text format: `<KERN><COLOR, 255, 255, 255>text<S>text<BR>text<END>`
- Text rendering is handled by SISLib (only ~25% decompiled)
- HSDRaw can read most but not all data file formats
- Source: werewolf.zip, ribbanya (2026-02-05)

## LLM/Agent Decompilation Notes

### Agent Scaffolding Approaches
Multiple contributors are running LLM agents for decompilation with different setups:
- **Claude Code with skills**: Using custom skill files that include m2c, decomp.me integration, and local build verification. One contributor's skills: https://github.com/lukechampine/melee/blob/claude-skills/.claude/skills/melee-decomp/SKILL.md
- **Embeddings database**: Using vector similarity search to find already-matched functions similar to the target, giving the agent examples to work from
- **File persistence**: Storing progress in files that survive context compaction to maintain state across long sessions
- **Pattern + script strategy**: Identify a pattern, write a script that captures it, then teach the agent to use the script as a guardrail
- Source: lukechampine, tsanummy, stephenjayakar (2026-02-03, 2026-02-09, 2026-02-10)

### Long Autonomous Sessions
- Opus 4.6 ran 11-24 hours autonomously on baselib functions
- Compaction alone is insufficient; after 3-4 compactions the agent loses initial context
- Persistent directives in files that survive compaction are essential
- One session took a 5KB function from 0% to 94% over ~24 hours of autonomous work
- $200/month Claude plan required; burns through weekly token limits quickly
- Source: tsanummy (2026-02-09, 2026-02-11)

### Codex Comparison
Codex is significantly slower than Claude thinking models (spends more tokens on reasoning) without proportionate improvement in quality. Not recommended for decompilation agent work.
- Source: itsgrimetime, tsanummy (2026-02-09)
