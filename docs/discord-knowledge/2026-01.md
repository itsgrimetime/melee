This document distills useful decompilation knowledge from Discord conversations in the #smash-bros-melee channel during January 2026.

## Compiler Patterns

### `s32` vs `int` Loop Counter Difference
If a loop generates a constant-compare-branch sequence like `li r5,0; cmpwi r5,0xf; bge`, the issue is likely `s32` vs `int` for the loop variable. Changing the loop counter from `s32` to `int` can fix unexpected codegen.
- Source: roeming (2026-01-14)

### Temp Variable to Split `fmadds` into `fmuls` + `fadds`
When you need separate `fmuls` and `fadds` instructions instead of a fused `fmadds`, introduce a temporary variable to break the expression:
```c
// fmadds (fused):
result = a * b + c;

// fmuls + fadds (separate):
float temp = a * b;
result = temp + c;
```
- Source: roeming, longlegschad (2026-01-15)

### `fcmpo` + `cror` Pattern Means `<=` or `>=`
When reading target assembly, `fcmpo` followed by `cror` is the compiler pattern for `<=` or `>=` float comparisons.
- Source: kellz (2026-01-08)

### Varargs and `crclr cr1eq` Instruction
The `crclr cr1eq` instruction appears in functions that call variadic functions (like `OSReport`). MWCC passes "are there any float arguments" as a condition flag in cr1. The variadic function checks this to skip saving float registers when no floats are passed.
```
crclr cr1eq       ; "no floats in arglist"
bl  OSReport      ; variadic function
```
The callee side saves every input register to the stack (can't know arg count at compile time) but skips the block of `stfd` instructions if cr1 indicates no floats.
- Source: altafen, lukechampine (2026-01-12)

### Condition Registers: cr0 vs cr1
PowerPC has 8 condition register fields (cr0-cr7):
- `cr0`: Used for integer comparisons
- `cr1`: Used for float comparisons and the varargs float optimization
- `cr2`-`cr7`: Rarely if ever used by MWCC
- Source: altafen (2026-01-13)

### Signed-to-Float Cast Assembly Pattern
When you see `xoris rX, rX, 0x8000` combined with `lis r3, 0x4330`, `stw` to stack, `lfd`, and `fsubs`, this is MWCC's pattern for casting `s32` to `f32`:
```c
// This C code:
item->xE34 = (f32)item->xCA0 * item->xCCC * attrs->x8;

// Generates the xoris + magic constant pattern
```
There are two variants: signed (uses `xoris`) and unsigned (different constants).
- Source: itsgrimetime, ieee802dot11ac, roeming (2026-01-25)

### Float Constants Go in `.sdata2`
When you write `float f = 1.0f;`, the compiler transforms it to a load from `.sdata2`:
```c
// You write:
float f = 1.0f;
// Compiler generates:
static const float sdata2_1 = 1.0f;
float f = sdata2_1;
```
Because loading from sdata2 is cheaper than materializing a float constant. Always write inline float constants rather than named `const float` variables unless there's a specific reason not to. Using a named const can produce different codegen.
- Source: altafen (2026-01-14)

### SDA Addressing: `.sbss` vs `.bss`
Small uninitialized data goes in `.sbss` (SDA-addressed), larger data goes in `.bss` (full 32-bit addressing). To move a symbol from `.sbss` to `.bss`, make it larger (e.g., by adding fields to its struct). The threshold is 8 bytes.

Reference table for data sections:
```
                  small      big
const             .sdata2    .rodata
initialized       .sdata     .data
uninitialized     .sbss      .bss
```
The wiki page https://github.com/doldecomp/melee/wiki/Data-sections is a useful reference.
- Source: altafen, blu9849, robojumper (2026-01-23)

### `& 0xFF` Mask and Type Conversions
The `& 0xFF` mask (generates `clrlwi`) is usually emitted automatically by `int` to `u8` conversions. Assume it is fake (an artifact of type mismatch) unless there's a specific reason it would be manually written. It can occasionally be real, e.g., to guard against wrapping.
- Source: rtburns, ribbanya (2026-01-24)

## Matching Techniques

### `GET_ITEM` Adds Stack Space
Using `GET_ITEM(gobj)` instead of `gobj->user_data` adds stack space. If a function has unexplained PAD_STACK, try using the GET_ITEM macro.
- Source: lukechampine (2026-01-09)

### `PAD_STACK` Is Usually a Sign of Missing Inlines
9 out of 10 times, needing `PAD_STACK` indicates stripped static inline functions that the compiler duplicated and removed. The remaining 1/10 is a developer forgetting to remove a variable (evidence: copy-pasted functions with few changes leaving unused vars).
- Source: revosucks (2026-01-10)

### Double Inline Pattern
When a function's stack or register allocation seems off, it may be because the original code had nested inlines:
```c
static void func_inline2(Item_GObj* gobj) {
    Item *item = GET_ITEM(gobj);
    Item_80268E5C(gobj, 1, ITEM_ANIM_UPDATE);
    item->entered_hitlag = efLib_PauseAll;
    item->exited_hitlag = efLib_ResumeAll;
}

static void func_inline(Item_GObj *gobj) {
    Item* item = GET_ITEM(gobj);
    it_80273454(gobj);
    func_inline2(gobj);
    item->xDD1_flag.b1 = 1;
}

void func(Item_GObj* gobj) {
    if (it_8027A09C(gobj) != 0) {
        func_inline(gobj);
    }
}
```
The compiler merges these into one function. The original TU likely had more functions than what appears in the assembly.
- Source: revosucks (2026-01-10, 2026-01-11)

### Declaration Order Affects Float Register Allocation
Reordering variable declarations (e.g., declaring `t` before `s`) can resolve float register swaps (f0 vs f2).
- Source: troy._ (2026-01-04)

### String Constants Should Be Inline
Use inline string constants in function calls, not references to a string table:
```c
// Correct:
func("abc");

// Wrong (fake):
func(&str_table[5]);
func(base + 0x70);
```
The compiler automatically combines string literals into a table and generates a base pointer load if needed.
- Source: altafen (2026-01-09, 2026-01-23)

### `base` Variable for Data Section Pointer
When `lis r3, symbol@ha; addi rN, r3, symbol@l` appears at the top of a function loading a pointer to the TU's data section, this is the compiler automatically loading a base pointer for multiple data references. Use constant strings directly and the compiler will generate the base pointer load automatically -- don't create a `base` variable.
- Source: altafen (2026-01-23)

### Chained NULL Assignment Pattern
Setting multiple fields to NULL in one statement can affect codegen:
```c
popup_data->text[0] = popup_data->text[1] = popup_data->text[2] = NULL;
```
Keep this pattern in mind if a tricky match involves reverse initialization order.
- Source: altafen (2026-01-23)

### `new_var = ptr` Alias Trick
Assigning a pointer to a new variable can help fix register alignment:
```c
new_var = jobj;  // alias helps register allocation
```
Also, splitting `Type* desc = (Type*)arg;` into separate declaration and assignment can affect codegen.
- Source: tsanummy (2026-01-23)

### Array Indexing vs `[0]` with Offset
Using `data->array[i]` generates `slwi` (shift left for index multiplication), while `data->array[0]` with manual offset does not. If you need a `slwi` in the output, use proper array indexing:
```c
// Generates slwi:
base->row_labels[i]

// Does NOT generate slwi:
base->row_labels[0] + offset
```
- Source: altafen (2026-01-24)

### Enum Min Flag
The `-enum min` flag was tested but ultimately concluded to not have been used globally for Melee. Some TUs seem to disprove it. Whether it was used per-TU remains untested.
- Source: rtburns (2026-01-01)

## Type Information

### `grGroundParam` Struct Layout
Partial layout of the stage ground parameter struct (size >= 0xDC):
```c
struct grGroundParam {
    f32 x0;          // stage scale
    s16 x4;          // shadow alpha
    u8 x6_pad[2];
    s16 x8;          // camera FOV
    s16 xA;
    s32 xC;          // cam dist min
    s32 x10;         // cam dist max
    s32 x14;         // tilt scale (degrees)
    f32 x18;         // vertical rotation
    f32 x1C;         // horizontal rotation
    f32 x20;         // "fixedness"
    f32 x24;         // bubble multiplier
    f32 x28;         // camera speed smoothness
    // ...
    s32 x30;         // pause min Z
    s32 x34;         // pause initial Z
    s32 x38;         // pause max Z
    f32 x3C;         // pause max angle up/down/left/right
    // ...
    bool x4C_fixed_cam;
    f32 x50-x64;     // fixed cam X/Y/Z, FOV, angles
    // ...0x6A-0xB0: item spawn weights region
    void* xB0;       // BGM data
    s32 xB4;         // number of BGM variants
    GXColor xB8-xD8; // 9 bubble colors (corners + edges)
};
```
- Source: righteousryan (2026-01-07)

### `MapLine` Struct Correction
```c
struct MapLine {
    u16 v0_idx;        // +0
    u16 v1_idx;        // +2
    s16 prev_id0;      // +4
    s16 next_id0;      // +6
    s16 prev_id1;      // +8
    s16 next_id1;      // +A
    u16 hi_flags;      // +C: collision face (top, bottom, right, left)
    u8 lo_flags;       // +E: interaction (dropthrough, ledgegrab)
    u8 mat_type;       // +F: material type of the line
};
```
Previously `lo_flags` was typed as `u16` and `mat_type` did not exist.
- Source: righteousryan (2026-01-09)

### `CollLineKind` Flags
```c
typedef enum CollLineKind {
    CollLine_Floor     = 1 << 0,
    CollLine_Ceiling   = 1 << 1,
    CollLine_RightWall = 1 << 2,
    CollLine_LeftWall  = 1 << 3,
} CollLineKind;

#define LINE_FLAG_KIND     (0b1111)
#define LINE_FLAG_EMPTY    (1 << 7)
#define LINE_FLAG_PLATFORM (1 << 8)
#define LINE_FLAG_LEDGE    (1 << 9)
#define LINE_FLAG_ENABLED  (1 << 16)
#define LINE_FLAG_HIDDEN   (1 << 18)
```
- Source: gigabowser (2026-01-09)

### Per-Article Attribute Structs
`item->xC4_article_data->x4_specialAttributes` is typed as `void*` but is actually a per-article struct. Create a new struct for each item type based on inferred field accesses:
```c
typedef struct itThingsAttributes {
    u8 x0[0x18];
    int x18;
} itThingsAttributes;
```
These structs are defined in `it/itCommonItems.h`. They don't appear to be shared across items.
- Source: werewolf.zip (2026-01-18)

### Struct Padding Trick
Subtraction-based padding for struct members to avoid manual offset math:
```c
typedef struct itLugiaAttributes {
    u8 x0[0x8];
    f32 x8;
    f32 xC;
    u8 _10[0x18 - 0x10]; // padding trick
    f32 x18;
} itLugiaAttributes;
```
- Source: kellz (2026-01-20)

### `ItemStateTable` Function Pointer Types
The `collided` field in `ItemStateTable` expects `HSD_GObjPredicate` (returns `bool`), not `void`. If a Coll function wraps a bool-returning function, it should also return bool:
```c
// Wrong:
void itFoo_Coll(Item_GObj* gobj) {
    it_8027A118(gobj, callback);
}

// Correct:
bool itFoo_Coll(Item_GObj* gobj) {
    return it_8027A118(gobj, callback);
}
```
- Source: blu9849 (2026-01-29)

## Function-Specific Insights

### Fighter Input Pipeline
- `HSD_PadCopyStatus`: Copy buffer from `HSD_PadMasterStatus`, used for most menus
- `HSD_PadGameStatus`: Used in-game, written to primarily in `Fighter_Spaghetti_8006AD10`
- `Fighter->input` is modified in `Fighter_Spaghetti_8006AD10`
- Source: werewolf.zip (2026-01-09)

### Ledge Grab Box Code
The ledge snap box calculation is at the start of `mpColl_80044164` (right side) and `mpColl_800443C4` (left side).
- Source: gigabowser (2026-01-09)

### `ftAnim_8006F994` Copy-Paste Pattern
A stack padding mystery was solved by recognizing that a function was lazily copy-pasted from `ftAnim_8006F994`. The developer copied the function, reworked it to accept the index as a parameter, but forgot to remove the original local `i` variable -- explaining the extra stack space.
- Source: revosucks (2026-01-07)

### Developer Debug Patterns
Several patterns in the codebase are likely artifacts of removed debug code:
- Checking a boolean before setting it to false (probably had a debug print warning)
- Switch cases with only 1 case
- Assert macros like `__assert("ftfoxspecialn.c", 305, "0")` with "Caution!!!" messages
- These are especially common in Fighter code, which had the most late-development debugging
- Source: werewolf.zip, antidote6212 (2026-01-07)

### `lb_8000B09C` / `it_80272C6C`
`it_80272C6C` calls `lb_8000B09C` which checks if any child node in a JObj hierarchy has an active animation. Returns false if all animations are still running (i.e., looping).
- Source: itsgrimetime (2026-01-27)

## Tool Tips

### Permuter Setup for Melee
kellz maintains a Melee-specific branch of decomp-permuter: https://github.com/sadkellz/decomp-permuter/tree/melee

Setup:
1. Create `nonmatchings` folder in decomp repo root
2. Place `permuter_settings.toml` and `prelude_ppc.inc` into it
3. If no devkitpro, add `--binutils build/binutils` to `configure.py` and set `assembler_command` in toml to `build/binutils/powerpc-eabi-as -mgekko`
4. Add `build/binutils` to PATH (permuter needs objdump)
5. Run from Linux/WSL:
```bash
./import.py "/melee/src/melee/<folder>/<file.c>" --function <func_name>
./permuter.py /melee/nonmatchings/<func_name>/ -j 20 --best-only --stack-diff
```
- Source: kellz, gigabowser (2026-01-05, 2026-01-07)

### decomp.me Float Values Must Be Verified
Float values in `.sdata2` are not verified by the matching algorithm. A function can show 100% match on decomp.me while having completely wrong float constants. Always check float values in objdiff (hover over the symbol) or compare against the assembly file.
- Source: werewolf.zip (2026-01-15)

### decomp.me vs Local Match Discrepancies
Functions can show 100% on decomp.me but not match locally due to:
- Different context (headers/types)
- Data/sdata addressing differences
- The decomp.me diffing algorithm ignoring exact data offsets

For linked files, run `ninja apply` after marking the file as Matching in `configure.py` to auto-fix reference issues.
- Source: werewolf.zip, ribbanya (2026-01-22, 2026-01-30)

### Dolphin Debugger
- Enable via Options > Configuration > Interface tab > "Enable Debugging UI"
- Memory tab: enter an address to see values in real-time
- Right-click a value > "Toggle Breakpoint" to pause when modified
- Set breakpoints at function start and check call stack to find callers
- Red register values mean "changed since last pause"
- Source: gigabowser, roeming (2026-01-09, 2026-01-16)

### `HSD_JObjGetChild` Inline
The pattern of checking jobj for NULL then accessing `jobj->child` is the `HSD_JObjGetChild` inline. Use it directly instead of manual null-check + child access, and use it inline without a temp variable:
```c
// Instead of:
HSD_JObj* bone;
if (jobj == NULL) { bone = NULL; } else { bone = jobj->child; }

// Use:
bone = HSD_JObjGetChild(jobj);
// Or even inline without temp
```
- Source: altafen (2026-01-16)

## Pitfalls and Gotchas

### AI-Generated Hallucinated Data
LLM-assisted decomp can produce hallucinated:
- **String literals**: e.g., "ViWait0801" instead of the actual "Vi0801.dat"
- **Float constants**: Completely wrong float values that still produce 100% instruction match (floats are in .sdata2, not verified by instruction diff)
- **Function/variable names**: Plausible but wrong identifiers

Always verify strings against the actual data section, check float values in objdiff, and ensure names have justification.
- Source: werewolf.zip (2026-01-15, 2026-01-16)

### Fake Match Red Flags
Signs of a fake match that should be scrutinized:
- Pointer arithmetic / casts that don't make semantic sense
- `PAD_STACK` hiding missing inlines
- `M2C_FIELD` usage without understanding the underlying struct
- `base + offset` for string table accesses instead of inline string constants
- Interleaved assignments purely to force instruction ordering
- `* 0` tricks to force register loads

Fake matches are sometimes merged with a `//todo fake` comment, but the preference is to find the real match or at least understand why the fake is necessary.
- Source: werewolf.zip, kellz, altafen (2026-01-23)

### TU Splits and Global Data
When data seems to be referenced from a different file:
- Check if the data is the last entry in a file's section
- Check if subsequent variables are only referenced in the other file
- Global data is almost entirely limited to the first file of a section or unsplit code
- `mn` and `vi` modules have some known unsplit globals
- Strings in `.data`/`.sdata` are probably not global (despite being string literals)
- dtk makes local symbols global with address appended if referenced from another TU
- Source: werewolf.zip, gamemasterplc (2026-01-10)

### `SpawnItem` Stack Struct
When an item-spawning function has a large local struct on the stack, it is likely `SpawnItem` -- a parameter struct for item instantiation. Use `--stack-structs` flag with m2c locally to get the struct layout. Most item creation functions pass this to `Item_80268B18`.
- Source: revosucks, ribbanya (2026-01-31)

## AI/LLM Decompilation Notes

### Three Independent LLM Toolchains Emerged
Three contributors independently built Claude Code-based decompilation pipelines:
1. **lukechampine (nemo)**: Skills-based approach with subsystem-specific examples, opcode-sequence duplicate finding. 780M tokens / $528.50 equivalent for all January work.
2. **itsgrimetime**: Parallel worktree agents, pre-commit hooks for quality, Dolphin debugger integration for validation. Skills repo: https://github.com/itsgrimetime/melee-decomp/tree/main/.claude/skills
3. **tsanummy**: Embedding-based similarity search, permuter integration, local decomp.me, autonomous "vacuum" runs (42 functions in 3.5 hours unattended). Repo: https://github.com/malvarezcastillo/melee-decomp-agent/
- Source: various (2026-01-09 through 2026-01-28)

### Key LLM Findings
- Models are good at trying many plausible C variations quickly to match ASM
- Main failure modes: pointer arithmetic fake matches, non-C89 code, wrong extern placement, excessive PAD_STACK, hallucinated names/strings/floats
- Per-subsystem example prompts significantly improve quality (e.g., item-specific skill with worked examples)
- The bottleneck is encoding human decompilation knowledge into a form LLMs can use
- Models struggle most with: sticking to conventions, avoiding fake matches, and multi-tool coordination
- "90% of my interaction is telling them to stop re-introducing pointer arithmetic" (itsgrimetime)
- Source: lukechampine, itsgrimetime, tsanummy (2026-01-09 through 2026-01-28)

### Fake Match Concept Defined
A "fake match" is C code that produces matching assembly but is obviously not the original source code. It games the match metric without contributing understanding. Examples: raw pointer arithmetic instead of proper structs, meaningless casts, `* 0` tricks. The match % is a proxy for understanding the codebase -- faking it defeats the purpose.
- Source: rtburns, itsgrimetime (2026-01-09)
